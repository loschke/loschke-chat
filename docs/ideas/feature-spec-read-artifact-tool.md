# Feature Spec: `read_artifact` Tool

## Kontext & Problem

Wenn externe Tools (z.B. `youtube_analyze`, `extract_branding`) Artifacts generieren, erhält das LLM nur Metadaten zurück:

```json
{
  "artifactId": "tEo-L-RsUIOQ",
  "title": "AI Transformation Insights",
  "type": "markdown",
  "version": 1
}
```

Der eigentliche Inhalt landet im Side Panel (UI), aber **nicht im Konversationskontext des LLMs**. Das verhindert, dass das LLM auf extern generierte Inhalte reagieren, kommentieren oder weiterverarbeiten kann.

---

## Ziel

Ein neues Tool `read_artifact` einführen, mit dem das LLM den Inhalt eines bereits gespeicherten Artifacts **explizit und opt-in** aus der Datenbank abrufen kann — und eine **Auto-Detection Logic**, die das für den User vollständig unsichtbar macht.

---

## Tool-Definition

```typescript
{
  name: "read_artifact",
  description: "Reads the content of a previously created artifact by its ID. Use this when you need to process, comment on, or respond to the content of an artifact that was generated by an external tool (e.g. youtube_analyze, extract_branding). Only call this when the user explicitly wants you to work with that artifact's content.",
  parameters: {
    artifactId: {
      type: "string",
      description: "The artifact ID to read, as returned by the tool that created it"
    },
    reason: {
      type: "string", 
      description: "Brief explanation of why the artifact content is needed"
    }
  }
}
```

---

## Erwarteter Return-Wert

```typescript
{
  artifactId: string,
  title: string,
  type: "markdown" | "html" | "code",
  language?: string,
  version: number,
  content: string,        // Der vollständige Artifact-Inhalt
  createdAt: string,      // ISO timestamp
  tokenCount?: number     // Optional: Größenhinweis für Context-Management
}
```

---

## Implementierungshinweise

### Backend

- Artifacts sind bereits in der DB gespeichert (sie werden für das Side Panel geladen)
- Der neue Endpoint braucht nur einen **Lesepfad** von `artifactId → content`
- Zugriffskontrolle: Das LLM darf nur Artifacts der **aktuellen Konversation** lesen (keine Cross-Conversation-Leaks)

```
GET /api/artifacts/:artifactId/content
Authorization: Bearer <session-token>
Constraint: artifactId must belong to current conversationId
```

### Sicherheit & Datenschutz

- **Scoped auf Konversation** — kein Zugriff auf Artifacts anderer Sessions
- **Kein automatischer Abruf** — das LLM muss das Tool explizit aufrufen (opt-in)
- **Audit-Log** empfohlen: wann hat das LLM welches Artifact gelesen

### Context-Window-Management

- Bei großen Artifacts (z.B. lange Transkripte) sollte der Return-Wert ein `tokenCount` mitliefern
- Das LLM kann dann entscheiden, ob es den vollen Inhalt oder nur einen Ausschnitt anfordert
- Optional: `read_artifact` um Parameter `maxTokens` oder `excerpt: true` erweitern

---

## Auto-Detection Logic

Der entscheidende Mechanismus damit `read_artifact` für den Nutzer vollständig unsichtbar bleibt.

### Konversations-Tracking

Das LLM muss im Konversationsverlauf für jedes Artifact festhalten, ob der Inhalt bekannt ist:

```typescript
type ArtifactMeta = {
  artifactId: string,
  title: string,
  type: string,
  source: "self" | "external",  // self = LLM hat es erstellt, external = Tool hat es generiert
  contentKnown: boolean,         // false bei external tools
  lastReferencedAt: number       // Timestamp der letzten Erwähnung im Chat
}
```

### Trigger-Bedingungen für automatischen read_artifact Aufruf

Das LLM ruft `read_artifact` **automatisch und ohne User-Hinweis** auf, wenn:

1. Der User auf ein Artifact Bezug nimmt (explizit per Titel oder implizit per Kontext)
2. `contentKnown === false` für dieses Artifact
3. Die Antwort des LLMs den Inhalt erfordert

```
User-Nachricht eingetroffen
        ↓
Bezieht sich der User auf ein Artifact?
        ↓ ja
contentKnown === false?
        ↓ ja
→ read_artifact() automatisch aufrufen [für User unsichtbar]
→ Inhalt in Kontext laden
→ Normal antworten ✅
```

### Implizite Bezüge erkennen

Der User wird selten sagen *"lies Artifact abc123"* — typische implizite Formulierungen:

| User sagt | LLM erkennt |
|---|---|
| "Was denkst du darüber?" | Bezug auf letztes Artifact |
| "Fass das anders zusammen" | Bezug auf letztes Artifact |
| "Gibt es da Widersprüche?" | Bezug auf letztes Artifact |
| "Schreib das als LinkedIn Post" | Bezug auf letztes Artifact |
| "Und das Video von vorhin?" | Bezug auf früheres Artifact per Kontext |

### System-Prompt Ergänzung (finale Version)

```
## Artifact Content Awareness

Nach jedem Tool-Aufruf merke dir, ob du den Artifact-Inhalt kennst:
- Artifacts die du mit create_artifact / create_review erstellt hast → Inhalt bekannt
- Artifacts die externe Tools generiert haben (youtube_analyze, extract_branding etc.) → Inhalt NICHT bekannt

Wenn der User auf ein Artifact Bezug nimmt dessen Inhalt dir nicht bekannt ist:
1. Rufe read_artifact automatisch auf — BEVOR du antwortest
2. Erwähne diesen Schritt NICHT explizit gegenüber dem User
3. Antworte dann mit vollem Kontext als wärst du von Anfang an informiert gewesen

Ziel: Der User soll keinen Unterschied merken zwischen eigenen und externen Artifacts.
```

### UI-Empfehlung: Subtiler Transparenz-Indikator

Auch wenn der Flow für den User unsichtbar sein soll, empfiehlt sich ein **minimaler Hinweis** — z.B. ein kleines Icon im Artifact-Header:

```
📄 AI Transformation Insights  [🔍 von KI gelesen]
```

Das wahrt Vertrauen ohne den Flow zu unterbrechen.

## Multi-Artifact-Handling

Sobald mehrere Artifacts in einer Konversation existieren, muss das LLM das **richtige** identifizieren, bevor es `read_artifact` aufruft.

### Das Problem

Eine Konversation kann beliebig viele Artifacts enthalten — z.B.:
- Artifact A: YouTube-Video Analyse (`external`)
- Artifact B: Branding-Extraktion einer Website (`external`)
- Artifact C: Selbst erstellter Report (`self`, Inhalt bekannt)

User sagt: *"Kannst du das nochmal umformulieren?"* — welches ist gemeint?

### Auflösungs-Hierarchie

Das LLM arbeitet die folgenden Stufen der Reihe nach ab:

**Stufe 1 — Explizite Nennung** (eindeutig, sofort auflösbar)
> "Die Branding-Analyse von vorhin" / "Das YouTube-Video" / "Der Report"
→ Direkter Match auf Titel, Typ oder Tool-Herkunft → `read_artifact()` ✅

**Stufe 2 — Zeitliche Nähe** (letztes referenziertes Artifact gewinnt)
> "Fass das anders zusammen"
→ Das Artifact mit dem höchsten `lastReferencedAt`-Wert → `read_artifact()` ✅

**Stufe 3 — Thematischer / semantischer Match**
> "Was sind die Kernfarben?" → passt zu Branding, nicht zu YouTube
> "Was waren die Key Takeaways?" → passt zu Video-Analyse
→ LLM wählt den semantisch naheliegendsten Kandidaten → `read_artifact()` ✅

**Stufe 4 — Disambiguation** (nur wenn alle anderen Stufen versagen)
> Zwei gleichwertige Kandidaten, kein klarer Kontext
→ LLM fragt kurz nach, statt zu raten:
> *"Meinst du die Video-Analyse oder den Branding-Report?"*

### Disambiguierungs-Flowchart

```
Mehrdeutiger User-Bezug erkannt
        ↓
Candidates = alle Artifacts im Konversationsverlauf
        ↓
Explizite Nennung erkannt?           → read_artifact(match) ✅
        ↓ nein
Nur 1 Candidate mit contentKnown=false?  → read_artifact(candidate) ✅
        ↓ nein
lastReferencedAt eindeutig?          → read_artifact(latest) ✅
        ↓ nein
Thematischer Match eindeutig?        → read_artifact(match) ✅
        ↓ nein
→ Kurze Nachfrage an den User
```

### Kandidaten-Scoring (optional, für höhere Präzision)

Für komplexe Konversationen kann ein einfaches Scoring-Modell eingesetzt werden:

```typescript
type ArtifactScore = {
  artifactId: string,
  score: number   // 0–100
}

function scoreCandidate(artifact: ArtifactMeta, userMessage: string): number {
  let score = 0
  if (titleMatch(artifact.title, userMessage))        score += 40  // Explizite Nennung
  if (artifact.lastReferencedAt === mostRecent)       score += 30  // Zeitliche Nähe
  if (semanticMatch(artifact.type, userMessage))      score += 20  // Thematik
  if (artifact.source === "external")                 score += 10  // Externer Inhalt unbekannt → wahrscheinlicher Kandidat
  return score
}

// Schwellenwert: score >= 60 → read_artifact direkt
// score < 60 mit mehreren Kandidaten → Nachfragen
```

### Mehrere Artifacts gleichzeitig lesen

In manchen Fällen bezieht sich der User auf **mehrere** Artifacts:
> "Vergleiche die Video-Analyse mit dem Branding-Report"

Das LLM soll dann **mehrere `read_artifact`-Calls parallel** ausführen:

```typescript
// Parallel-Aufruf, nicht sequenziell
const [videoContent, brandingContent] = await Promise.all([
  read_artifact({ artifactId: "abc123" }),
  read_artifact({ artifactId: "xyz789" })
])
```

**Context-Window-Hinweis:** Bei mehreren großen Artifacts gleichzeitig sollte das LLM auf `excerpt: true` zurückfallen und nur die relevanten Abschnitte laden.

### System-Prompt Ergänzung: Multi-Artifact

```
## Multi-Artifact-Auflösung

Wenn mehrere Artifacts in der Konversation existieren und der User-Bezug unklar ist:

1. Explizite Nennung im Text → dieses Artifact wählen
2. Kein klarer Hinweis → das zuletzt referenzierte Artifact wählen
3. Thematischer Kontext passt eindeutig zu einem Artifact → dieses wählen
4. Immer noch unklar → kurz nachfragen, NICHT raten

Bei Vergleichsanfragen ("vergleiche X mit Y") → beide Artifacts parallel lesen.
Achte bei mehreren großen Artifacts auf das Context-Window — nutze excerpt: true wenn nötig.
```

---

## Gewünschtes Verhalten des LLMs

Das LLM soll `read_artifact` aufrufen, wenn:
- Ein externes Tool ein Artifact erstellt hat (erkennbar an der `artifactId` im Tool-Result)
- Der User fragt, das LLM darauf zu reagieren, es zu kommentieren oder weiterzuverarbeiten
- **Nicht** automatisch bei jeder Artifact-Erstellung — nur bei explizitem Bedarf

### Beispiel-Flow (Single Artifact)

```
1. User: "Fasse dieses YouTube-Video zusammen"
2. LLM ruft youtube_analyze auf → erhält artifactId: "abc123"
3. Artifact erscheint im Side Panel
4. User: "Was denkst du darüber?"
5. LLM erkennt: Bezug auf "abc123", contentKnown = false
6. LLM ruft read_artifact({ artifactId: "abc123" }) automatisch auf [unsichtbar]
7. LLM antwortet mit vollem Kontext ✅

User merkt: nichts. Einfach eine Antwort.
```

### Beispiel-Flow (Multi Artifact)

```
1. User: "Analysiere das Video und extrahiere das Branding von ihrer Website"
2. LLM ruft youtube_analyze → artifactId: "abc123"
3. LLM ruft extract_branding → artifactId: "xyz789"
4. User: "Vergleiche beide — passen Botschaft und Marke zusammen?"
5. LLM erkennt Vergleichsanfrage → beide Artifacts relevant
6. LLM ruft parallel auf:
   - read_artifact({ artifactId: "abc123" })
   - read_artifact({ artifactId: "xyz789" })
7. LLM antwortet mit integriertem Vergleich ✅
```

---

## Optionale Erweiterung: Partial Read

Für sehr große Artifacts (z.B. Video-Transkripte > 10.000 Wörter):

```typescript
read_artifact({
  artifactId: "abc123",
  excerpt: true,        // Nur ersten ~500 Tokens
  startToken: 1000,     // Pagination möglich
  maxTokens: 2000
})
```

## Offene Fragen

1. Soll das Tool auch für `create_artifact`-Inhalte nutzbar sein, oder nur externe Tools? (Eigentlich unnötig, da LLM den Inhalt kennt)
2. Soll der User sehen, wenn das LLM ein Artifact liest? (Subtiler Transparenz-Indikator im Artifact-Header?)
3. Versionierung: Soll immer die neueste Version gelesen werden, oder eine spezifische?
4. Was passiert bei sehr großen Artifacts — hard limit oder Pagination via `partial read`?
5. Soll das Scoring-Modell für Artifact-Auflösung konfigurierbar sein, oder reichen feste Schwellenwerte?
